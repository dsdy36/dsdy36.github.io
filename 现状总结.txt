向您汇报我目前在自动反馈生成方向上的研究进展和遇到的问题。

一、在做什么

基于 TAFFIES 论文（Pike, Lee & Towey, 2022）的方法论，搭建了一条自动化的反馈签名（Feedback Signature, FS）生成管线。

工作流程由两部分组成：

1. CW 端（训练数据生成）：用 AI 生成 50 个模拟学生提交，每个学生的 README 精确标注被注入了哪个错误或正确 pattern。例如 "use f-string ORDER BY without whitelist"。这一步的目的是确保有可靠的 ground truth。

2. FS 端（反馈签名生成）：读取 50 个 README 中的 pattern label，按文本相似度将学生分组（例如所有"用了 f-string ORDER BY 但没做验证"的学生归为一组）。每个组发给 DeepSeek，AI 同时看到该组所有学生的完整代码和 label 描述，生成一条 check function——一个检测代码中是否存在特定模式的 Python 函数，例如 "检查是否使用了 f-string 拼接 SQL 但没有 whitelist"。未覆盖的学生通过 FCC（覆盖检查）迭代补充，最多 3 轮。

同时，我搭建了一个行为沙箱：在 subprocess 中 mock 掉 sqlite3 和 Flask 依赖，向学生函数注入恶意输入（如把排序列名设为 "1; DROP TABLE Playlist--"），观察 SQL 注入是否真的成功了。这个方法在 4 个可测试的标准上跑了 200 次，全部成功完成，能够干净地将学生分为 VULNERABLE 和 SAFE 两组，零歧义。


二、当前结果

在全量 14 个评分标准 × 50 个学生上，盲测结果如下：

  Precision = 0.15, Recall = 0.62, F1 = 0.24
  正确检出 51 个错误，误报 300 个，漏报 31 个

14 个标准中，只有 2 个勉强可用：RQ1_4 (F1=0.75) 和 RQ1_3 (F1=0.50)。其他 12 个的 F1 均在 0.00-0.37 之间。300 个误报中 88% 来自 9 条过于宽泛的 check function。


三、为什么是根本性问题

以 RQ2_3 "ORDER BY 需要 whitelist 验证以防止 SQL 注入"为例说明。

两个学生的代码在 SQL 拼接部分完全一样：

  S005（有漏洞）：
    query += f" ORDER BY {sort_column} {sort_order}"
    cursor.execute(query)

  S002（安全）：
    if sort_column not in ALLOWED:       # 多了这段验证
        sort_column = 'p.Name'
    query += f" ORDER BY {sort_column} {sort_order}"
    cursor.execute(query)

两条代码都用了 f-string 拼接 ORDER BY。区别仅仅是前面有没有验证逻辑。但验证逻辑的写法是开放的——dict.get()、if/elif 链、set membership、函数封装——每一种都是合理的。AI 生成的 check function 只能用几个固定的正则去匹配这个开放集合，结果必然是：漏掉某些没见过的写法（漏报），或者把写法不同但实际做了验证的代码标为错误（误报）。

根本矛盾：错误本质是"没有阻止 SQL 注入"——这是代码的行为属性，无法从固定的语法特征中推导。14 个评分标准中 13 个都属于这类"缺失行为"的问题。


四、目前的困境

行为沙箱已经证明可以完美解决这个问题：直接执行代码，注入恶意输入，观察结果。不需要关心验证是怎么写的，只关心 SQL 注入是否真的被阻止了。

但行为测试目前是一个外部脚本（batch_runner.py），不在 FS registry 里。FS registry 里的 check function 签名是 def check(code: str) -> tuple，只能读文本，不能执行。

我尝试让 AI 生成内嵌沙箱执行能力的 check function（即在函数内部构造 mock 环境 + 恶意输入 + subprocess 执行）。结果 4 个标准中 3 个 FAILED——AI 写不好完整的 mock 层，总是漏掉某个依赖（get_db_connection()、Path.mkdir()、全局变量引用等），导致 check function 崩溃或返回错误结果。

目前的三个路径各有困境：
  - 继续纯语法的 check function：F1 天花板约 0.3，已经触达
  - 让 AI 生成内嵌沙箱的 check function：可靠性不足（4/1 成功）
  - 人手写行为测试模板 + AI 做模板匹配：可行但 AI 角色退化

想请教您的意见：是否应该接受"人手写模板 + AI 匹配"的务实方案，还是有其他方向值得探索？
